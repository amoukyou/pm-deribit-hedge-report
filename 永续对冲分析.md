# 永续合约对冲 PM Binary 分析

## Deribit 永续合约成本

| 项目 | 费率 |
|------|------|
| Perp Maker | **-0.025%** (rebate) |
| Perp Taker | 0.075% |
| Funding Rate | 8h 结算，cap ±0.5% (BTC) |
| 期权 Taker | 0.04% of underlying |

---

## 核心问题：Binary 是阶跃函数，Perp 是线性

PM binary payoff: 到期时 BTC > K → $1, 否则 → $0
Perp payoff: 线性，ΔPnL = position × (S_exit - S_entry)

**不能用静态 perp 头寸对冲 binary。** 必须动态调整。

### Binary Option 的 Delta

对于 digital call（BTC > K at time T）：
```
Δ_binary = φ(d2) / (S × σ × √T)
```
其中 φ 是标准正态 PDF，d2 = (ln(S/K) + (r - σ²/2)T) / (σ√T)

关键特征：
- **远离 K**：Δ ≈ 0（结果已确定）
- **接近 K**：Δ 很大且波动剧烈
- **T→0 且 S≈K**：Δ → ∞ （pin risk，gamma 爆炸）

## 触发式动态平仓策略（Zone Hedging）

### 设计思路

将价格空间分为三个区域：

```
    ←── 确定输区 ──→|←── 活跃对冲区 ──→|←── 确定赢区 ──→
                     K - d            K            K + d
```

| 区域 | 条件 | 操作 | Binary 价值 |
|------|------|------|------------|
| 确定输 (Deep OTM) | S < K - d | 平掉 perp，接受 binary→0 | ≈ 0 |
| 活跃对冲 | K-d < S < K+d | 持续 delta hedge | 0.2 ~ 0.8 |
| 确定赢 (Deep ITM) | S > K + d | 平掉 perp，接受 binary→1 | ≈ 1 |

**d 的选择** = f(σ, T_remaining)，例如 d = 2σ√T × S（2 个标准差）

### 具体流程

1. **开仓**：PM 上 YES 被成交于价格 p → 立即在 Deribit 开 Δ=p 的 perp 空头
2. **活跃区维护**：每隔 Δt 重新计算 delta，调整 perp 仓位
3. **触发平仓条件**：
   - 上触发（S > K + d）：binary ≈ 1 → 平掉 perp → PnL = (1-p) - perp_loss + rebate
   - 下触发（S < K - d）：binary ≈ 0 → 平掉 perp → PnL = -p + perp_gain + rebate
   - 时间触发：临近到期（T < T_min）且在活跃区 → 停止对冲，接受结果
4. **到期结算**：PM binary 自动结算

### 触发距离的优化

触发距离 d 是核心参数：
- **d 太大**：几乎不触发，一直在活跃区 → rebalance 成本高
- **d 太小**：频繁触发 → 可能在触发后价格回到活跃区（whipsaw）

建议初始设定：
- 日线市场：d ≈ 1.5-2% of S（BTC ~$85k → d ≈ $1300-$1700）
- 周线市场：d ≈ 3-5% of S
- 随 T 减小收窄 d（因为随着到期临近，越远的价格也变确定）

## Perp vs Options 对冲对比

| 维度 | 永续合约 (Perp) | 期权 (Call Spread) |
|------|----------------|-------------------|
| 适用时间框架 | **所有**（5min~周线）| 日线/周线（需匹配到期） |
| 前期成本 | 低（只需保证金） | 高（支付 premium） |
| 持续成本 | rebalance 手续费 + funding | 无（静态持有） |
| Pin risk | **高**（核心风险） | 低（已在 spread 中定价） |
| 操作复杂度 | 高（需自动化） | 低（开仓即完成） |
| 基差风险 | 取决于 rebalance 频率 | 取决于 ΔK 宽度 |
| Deribit 费用 | maker -0.025% (rebate!) | taker 0.04% |
| 极端行情 | 灵活应对 | 锁死 |

### 结论：取决于时间框架

| PM 市场类型 | 推荐对冲工具 | 原因 |
|------------|-------------|------|
| 5min / 15min | Perp（或不做） | 无对应期权到期；gamma 极高，风险大 |
| 1H / 4H | **Perp + 触发平仓** | 时间够 rebalance，无完美期权匹配 |
| 日线 | Perp 或 期权均可 | 期权匹配度好；perp 更灵活 |
| 周线 | **期权优先** | 静态对冲省心；期权流动性好 |

## Perp 对冲的成本模型

### Rebalancing 成本
每次调仓成本 ≈ |ΔPosition| × taker_fee(0.075%) 或 maker_fee(-0.025%)

如果用限价单（maker）：**每次调仓还能拿 rebate！**

### Funding 成本/收入
- 8h 结算一次
- 牛市中 funding 通常为正 → 空头（做空对冲）**收取** funding
- 这可能是额外收入来源！

### 总成本估算（日线市场为例）

假设：BTC=$85000, K=$85000, σ=60% annualized, T=1天
- Binary delta at-the-money ≈ 0.5
- 预计 rebalance 次数 ≈ 10-20 次/天
- 每次调仓量 ≈ 0.05-0.1 BTC
- 如果全用 maker 单：成本 ≈ 0（甚至正收益）
- Funding：牛市空头收 ~0.01-0.03%/8h

### 理论 hedging cost
动态对冲 binary 的理论成本 = binary option 的公允价格
即：你 hedge 的成本 ≈ 你在 PM 上成交 binary 的价格
利润来源全在于：**rebate + spread + mispricing**

## 风险场景分析

### 场景1：价格远离 K（最优）
- Binary 快速归零或归一
- 触发平仓，rebalance 次数少
- 净收益 ≈ rebate + spread（几乎无对冲成本）

### 场景2：价格在 K 附近震荡（最差）
- Delta 剧烈波动，频繁 rebalance
- 手续费累积，gamma 成本高
- 可能亏损（对冲成本 > rebate + spread）

### 场景3：闪崩/暴涨穿越 K（中等）
- 快速穿越 → 触发平仓
- 可能有滑点损失
- 但如果速度够快，损失有限

## 实操考量

### 自动化架构
```
PM WebSocket (订单成交)
  → 计算 delta
  → Deribit API (调整 perp)
  → 监控触发条件
  → 触发平仓逻辑
```

### 关键参数
- rebalance 频率：每分钟 or 每次 PM 价格变动 >0.5%
- 触发距离 d：根据 σ 和 T 动态调整
- 最大单市场仓位：控制极端情况风险
- Delta 模型：BS digital delta or 经验模型
